## Findings — Infrastructure/Discovery/

### F-D-01 · HIGH · ExtensionDiscoveryService — Broad `\Exception` catches swallowing all errors

**File:** `src/Infrastructure/Discovery/ExtensionDiscoveryService.php`
**Lines:** 135, 234, 360, 413, 479
**Description:** Five separate catch blocks use `\Exception` or `\Throwable`. The catch at line 234 is intentional (fallback strategy), but the others at lines 135, 360, 413 and 479 swallow file I/O errors, include-errors, and JSON parse failures into a single generic log entry. A malformed `PackageStates.php` or permission error on `vendor/composer/installed.json` will silently produce an empty extension list with no actionable error.
**Fix direction:** Replace with specific exception types (`\JsonException`, `\RuntimeException`). Distinguish "no data found" (expected) from "read failed" (unexpected). Throw on unexpected errors or surface as ValidationIssue.

---

### F-D-02 · HIGH · ExtensionDiscoveryService — Bug #163: PackageStates called before Composer method, no v5 format guard

**File:** `src/Infrastructure/Discovery/ExtensionDiscoveryService.php`
**Lines:** 77–93
**Description:** `discoverFromPackageStates()` is called first (line 77), before `discoverFromComposerInstalled()` (line 93). In Composer-managed installations, `PackageStates.php` may be a v4-format file generated by the TYPO3 installer for legacy compatibility. The current code has no guard checking whether the PackageStates format version matches what is expected for the detected installation type. Extensions discovered from a stale v4 PackageStates in a Composer v5 installation are treated as authoritative, causing incorrect extension state.
**Fix direction:** When installation type is `COMPOSER_*`, either skip PackageStates parsing or validate that its format version matches expectations. `ExtensionDiscoveryService.determineInstallationType()` already distinguishes installation types — use this to gate the PackageStates call.
**Related:** GitHub issue #163, story `p2-4-fix-packagestates-v5-composer-extensions`

### F-D-03 · HIGH · ConfigurationDiscoveryService — No VersionProfileRegistry usage, hardcoded v11/v12 paths throughout

**File:** `src/Infrastructure/Discovery/ConfigurationDiscoveryService.php`
**Lines:** 163–164, 267, 278, 545, 558
**Description:** The class contains no reference to `VersionProfileRegistry`. All version-specific paths (`config/system/settings.php`, `typo3conf/LocalConfiguration.php`, `config/sites`, `typo3conf/sites`) are hardcoded inline. This contradicts the architecture pattern established by the VersionProfile system and will produce incorrect paths for future TYPO3 versions without a code change.
**Fix direction:** Inject `VersionProfileRegistry`, resolve path constants from the detected TYPO3 version's profile. Follow the pattern already used in `ComposerInstallationDetector` and `ExtensionDiscoveryService`.

## Findings — Infrastructure/ExternalTool/

### F-E-01 · HIGH · TerApiHttpClient — Incomplete HTTP status code handling, no retry logic

**File:** `src/Infrastructure/ExternalTool/TerApiHttpClient.php`
**Lines:** 99–117
**Description:** `isSuccessfulResponse()` only handles HTTP 500 (returns false) and HTTP 400 (assumes "not found"). No handling for: 401/403 (auth failure), 429 (rate limit — TER enforces rate limits), 503/504 (transient server errors). HTTP 500 returns `false` with no retry. A single transient server error permanently fails the request.
**Fix direction:** Expand status code handling to cover auth failures (throw) and transient errors (distinguish from permanent). Add at minimum one retry for 5xx responses with a short delay. Consider extracting a `RetryingHttpClient` decorator.

---

### F-E-02 · HIGH · TerApiClient — `\Throwable` catch converts all errors to `TerApiException`

**File:** `src/Infrastructure/ExternalTool/TerApiClient.php`
**Lines:** 68, 99
**Description:** Both `hasVersionFor()` and `getLatestVersion()` catch `\Throwable`. This means `TypeError`, `Error`, `OutOfMemoryError`, and logic errors in the response parser are all converted to `TerApiException`. Bugs in `TerApiResponseParser` or `VersionCompatibilityChecker` are permanently hidden.
**Fix direction:** Catch specific exceptions (`TerApiException`, `\JsonException`, `\RuntimeException`). Let `Error` subclasses propagate.

---

### F-E-03 · HIGH · AbstractGitProvider — `\Throwable` catch converts all errors to `GitProviderException`

**File:** `src/Infrastructure/ExternalTool/GitProvider/AbstractGitProvider.php`
**Lines:** 90–102
**Description:** Base `makeRequest()` catch block converts all non-`GitProviderException` errors (including network stack errors, JSON parse errors, OOM) to `GitProviderException`. Subclasses cannot distinguish infrastructure failures from API errors.
**Fix direction:** Catch `\RuntimeException` and network-specific exceptions. Propagate `Error` subclasses.

---

### F-E-04 · HIGH · GitVersionParser — Assumes main-branch compatibility implies all-tag compatibility

**File:** `src/Infrastructure/ExternalTool/GitVersionParser.php`
**Lines:** 41–45
**Description:** If the main branch `composer.json` is compatible, all non-prerelease tags are returned without per-tag validation. This is documented as a simplification (lines 30–32, 47–48) but produces false positives: an extension that added TYPO3 14 support on its main branch but released its last tag against TYPO3 12 will be reported as compatible for all tags.
**Fix direction:** Either validate `composer.json` per tag (expensive), or clearly document the limitation in the return value / method contract so callers can weight results accordingly. At minimum, add a `$reliability` field to results.

---

### F-E-05 · MED · GitHubClient — No GraphQL rate limit handling

**File:** `src/Infrastructure/ExternalTool/GitProvider/GitHubClient.php`
**Lines:** 353–382
**Description:** `graphqlRequest()` does not read `x-ratelimit-remaining` headers. Under sustained use (large installations with many extensions), the client will hit GitHub's rate limit and receive 403 responses, which are caught and wrapped without retry or backoff.
**Fix direction:** Read `x-ratelimit-remaining` and `x-ratelimit-reset` headers after each request. If remaining < threshold, sleep until reset or throw a rate-limit-specific exception the caller can handle.

---

### F-E-06 · MED · PackagistClient — HTTP error handling gaps, no retry

**File:** `src/Infrastructure/ExternalTool/PackagistClient.php`
**Lines:** 45, 52, 70, 77, 95
**Description:** Only HTTP 200 is treated as success. 429 (rate limit), 404 (package not found — expected), and 5xx are not explicitly handled. `ExternalToolException` is thrown on any non-200 without classification, making it impossible for callers to retry transient errors vs. handle permanent ones differently.
**Fix direction:** Distinguish 404 (return null/empty), 429 (retry with backoff), 5xx (retry once, then throw). Align error model with `TerApiClient`.

### F-E-07 · MED · TerApiClient — Internal dependencies constructed in constructor

**Description:** `TerApiHttpClient`, `TerApiResponseParser`, and `VersionCompatibilityChecker` are constructed inline. These cannot be mocked or replaced in tests. Unit tests for `TerApiClient` necessarily exercise all three collaborators.

### F-E-08 · MED · VersionCompatibilityChecker — Wildcard pattern parsing fragile

**File:** `src/Infrastructure/ExternalTool/VersionCompatibilityChecker.php`
**Lines:** 113, 119, 126
**Description:** `substr(..., 0, -2)` hardcodes `.*` suffix removal. `preg_match()` return value is not checked for `false` (regex error). If the input version string is malformed, silent non-match occurs.
**Fix direction:** Validate input patterns before processing. Check `preg_match()` return value explicitly.

## Findings — Infrastructure/Reporting/

### F-R-01 · HIGH · TemplateRenderer — No exception handling on Twig render calls

**File:** `src/Infrastructure/Reporting/TemplateRenderer.php`
**Lines:** 42, 46, 103, 108, 177, 182
**Description:** All `$this->twig->render()` calls are unguarded. A missing template file, a Twig syntax error, or an undefined variable in the template will propagate an unhandled `\Twig\Error\*` exception through `ReportService` and abort report generation entirely — including formats that were already generated successfully.
**Fix direction:** Wrap each render call in a try-catch for `\Twig\Error\Error`. Return a `ReportingResult::failure()` for the affected format. Other formats continue normally.

---

### F-R-02 · HIGH · ReportContextBuilder — Unsafe chained null-able calls and false/null confusion

**File:** `src/Infrastructure/Reporting/ReportContextBuilder.php`
**Lines:** 64, 87–88
**Description:**
- Line 64: `$r->getExtension()->getKey()` — if `getExtension()` returns null, fatal `TypeError`.
- Lines 87–88: `reset($installationDiscovery) ?: null` — `reset()` returns `false` on empty array, not `null`. Code that checks `=== null` will fail silently.
**Fix direction:** Add null check before chained call. Replace `reset() ?: null` with an explicit empty-array guard.

---

### F-R-03 · HIGH · ReportFileManager — Uncaught file write failures

**File:** `src/Infrastructure/Reporting/ReportFileManager.php`
**Lines:** 79, 103, 154
**Description:** `file_put_contents()` returns `false` on failure (disk full, permission denied). All three calls ignore the return value. A failed write produces no error; the report is silently incomplete.
**Fix direction:** Check return value; throw `\RuntimeException` on failure with path and error context.

---

### F-R-04 · MED · ReportService — `\Throwable` catch at format loop level

**File:** `src/Infrastructure/Reporting/ReportService.php`
**Line:** 86
**Description:** Catches `\Throwable` per format in the generation loop. Intent is to allow partial success (one format fails, others succeed), which is correct. However, `OOM` and `Fatal` errors are caught and treated as recoverable format failures. This will mask infrastructure-level failures.
**Fix direction:** Narrow to `\RuntimeException` and `\Twig\Error\Error`. Let `Error` subclasses propagate.

---

### F-R-05 · MED · ReportContextBuilder — Hardcoded fallback TYPO3 version `'13.4'`

**File:** `src/Infrastructure/Reporting/ReportContextBuilder.php`
**Line:** 85
**Description:** If target version is null, `'13.4'` is hardcoded as default. This will silently produce incorrect reports when the analyzer is run without a target version specified. The hardcoded string will also become wrong when TYPO3 14 becomes the default target.
**Fix direction:** Derive default from `VersionProfileRegistry::getLatestSupportedVersion()` or require the target version to be explicitly set before report generation.

---

### F-R-06 · MED · TemplateRenderer — Inconsistent format error handling

**File:** `src/Infrastructure/Reporting/TemplateRenderer.php`
**Lines:** 53, 117
**Description:** `renderMainReport()` throws `\InvalidArgumentException` for unsupported formats (line 53). `renderSingleExtensionReport()` returns `null` for unsupported formats (line 117). Caller cannot apply consistent handling.
**Fix direction:** Standardise: throw in both, or return null in both with a consistent contract.

---

### F-R-07 · LOW · ReportFileManager — Path traversal risk on filename concatenation

**File:** `src/Infrastructure/Reporting/ReportFileManager.php`
**Line:** 77
**Description:** `$outputPath . $renderedReport['filename']` — if `filename` is attacker-controlled or contains `../`, writes outside the intended output directory. In current usage the filename comes from the template renderer so risk is low, but the pattern is unsafe by design.
**Fix direction:** Sanitize filename: `basename()` or validate no `..` components before concatenation.

**Title:** `p2-x-reporting-hardening — Harden TemplateRenderer and ReportFileManager against silent failures`

5. All existing reporting tests pass; add tests for failed Twig render and failed file write scenarios.

**Severity:** Addresses F-R-01, F-R-02, F-R-03, F-R-04 (all HIGH/MED).

## Challenges

1. **Spike benchmark did not match production execution path** — the spike benchmarked Composer CLI commands directly in the shell. Production code used Symfony Process for subprocess execution. Symfony Process appeared to behave as if no Composer cache was available, producing dramatically worse performance (17-22s per call vs spike expectations). The root cause — whether cache, environment variable inheritance, or another factor — was never conclusively diagnosed. This led to four stories (2-1, 2-2, 2-5, 2-5a) being built on a flawed performance assumption.

2. **Architectural churn** — resolution approach changed three times within one epic. `ComposerSourceParser`, `PackagistVersionResolver`, `ComposerVersionResolver`, `ComposerEnvironment` were all created and then deleted or superseded within the same epic.

3. **Untested dependency** — `ComposerConstraintChecker` had zero test coverage. The compound constraint bug (#223) went undetected until Story 2-8 was added as a mid-sprint fix. The class was consumed by multiple stories without anyone flagging the coverage gap.

4. **No formal dependency coverage check** — story completion verified new code but not coverage on pre-existing classes the story relied on. The 80% coverage rule existed in project-context but was not enforced per-story on dependencies.

5. **Code duplication and class bloat** — multiple stories added private methods to `ExtensionDiscoveryService` and other classes without stepping back to evaluate structural health. Code reviews focused on correctness of new code, not impact on the modified class's overall structure.

6. **Epic 1 retrospective not persisted** — `epic-1-retrospective` is marked `done` in sprint-status but no document exists on disk. Action items from that retro could not be tracked or followed through.

## Key Insights

1. **Spike benchmarks must use the same execution mechanism as production code.** Shell performance does not predict Symfony Process performance. If production wraps a CLI tool in Symfony Process, the spike must benchmark through Symfony Process.

2. **Spike assumptions need early re-validation.** After the first implementation story that depends on spike findings, run a quick smoke test against a real installation before committing to the next story. The data for the better decision (git-native) existed in the spike but was not weighted correctly in the recommendation.

3. **Story completion must verify test coverage on dependencies, not just new code.** When a story introduces or relies on a class, the completion checklist should verify that class has adequate test coverage (minimum 80%). Flag untested dependencies before marking ready-for-review.

4. **Code reviews need a structural health lens.** Not just "does the new code work" but "is the class still well-structured after this change?" Flag duplication, growing responsibilities, and bloated classes during review.

5. **Epic sequencing should be driven by stakeholder priorities, not purely technical ordering.** Three stakeholder voices emerged during this epic that the current roadmap doesn't account for.
